# Transaction Processing Pipeline

An async backend that takes a dirty CSV of financial transactions, cleans it, runs anomaly detection, classifies uncategorised rows with an LLM, and generates a narrative spending summary. all in the background via Celery, so the upload endpoint responds immediately instead of making the client wait.

Built for the Alemeno Backend + DevOps internship assignment.

## Why it's built this way

The assignment data is intentionally messy: mixed date formats, dollar signs in amount fields, inconsistent casing, blank categories, duplicate rows, and a handful of transactions that are obviously too large to be normal. Doing all of that cleaning synchronously inside an HTTP request would mean the client sits there waiting while pandas chews through the file and the LLM gets called and if the LLM is slow or rate limited, the request just hangs. So the upload endpoint does the bare minimum (save the file, create a job row, hand it off to Celery) and returns in milliseconds. Everything else happens in a worker process that the client polls for status.

## Stack

- FastAPI: for the API layer
- PostgreSQL: for storage (jobs, transactions, summaries)
- Redis + Celery: as the job queue
- Gemini 2.5 Flash Llm: for category classification and the narrative summary (free tier)
- Docker Compose: to run all four services together

## Project structure

```
app/
  main.py              FastAPI entrypoint, mounts the jobs router
  config.py            Reads and validates all env vars in one place
  database.py          SQLAlchemy engine, session factory, get_db dependency

  models/               SQLAlchemy table definitions
    job.py
    transaction.py
    job_summary.py

  schemas/              Pydantic request/response shapes
    job.py

  routers/
    jobs.py              All four API endpoints

  services/              Pure business logic, no DB or HTTP calls baked in
    cleaner.py            Date/amount/casing normalisation, dedup
    anomaly.py             3x-median outlier + currency mismatch rules
    llm.py                 Gemini calls with retry/backoff

  worker/
    celery_app.py          Celery configuration
    tasks.py                 The actual pipeline, step by step

migrations/
  001_initial_schema.sql   Raw SQL run automatically by the postgres image on first boot
```

Services are kept free of database and Celery imports on purpose. `cleaner.py` and `anomaly.py` just take a DataFrame and return a DataFrame you could unit test them without spinning up Docker at all. `tasks.py` is the only place that wires services together and talks to the database.

## Running it

You need Docker and Docker Compose. Nothing else — no local Python install, no local Postgres.

```bash
git clone https://github.com/khushi0433/transaction-pipeline.git
cd transaction-pipeline
cp .env.example .env
```

Open `.env` and fill in `GEMINI_API_KEY`. Get a free one at [aistudio.google.com/apikey](https://aistudio.google.com/apikey) — no billing required, just a Google account. The other values in `.env.example` are fine to leave as-is for local use.

```bash
docker compose up --build
```

First run takes a few minutes — it's pulling the Postgres and Redis images and installing Python dependencies. Once it settles, you'll see the API log `Uvicorn running on http://0.0.0.0:8000` and the worker log `celery@... ready.`. Both need to show up; if the worker isn't there, jobs will sit on `pending` forever (Redis will hold the task, but nothing's listening).

Health check:
```bash
curl http://localhost:8000/health
```

Interactive API docs (auto-generated by FastAPI):
```
http://localhost:8000/docs
```

## API

### Upload a CSV
```bash
curl -X POST http://localhost:8000/jobs/upload \
  -F "file=@transactions.csv"
```
Returns immediately with a `job_id`:
```json
{"job_id":"439c0de4-ccfa-40cc-9c06-1f157856f450","status":"pending","message":"Job accepted. Poll /jobs/{job_id}/status for updates."}
```

### Check status
```bash
curl http://localhost:8000/jobs/439c0de4-ccfa-40cc-9c06-1f157856f450/status
```
Status moves `pending → processing → completed` (or `failed`). Once completed, the response includes a short `summary` block with totals and risk level so you don't have to fetch the full results just to know how it went.

### Get full results
```bash
curl http://localhost:8000/jobs/439c0de4-ccfa-40cc-9c06-1f157856f450/results
```
Returns the cleaned transaction list, the anomaly subset, and the LLM-generated narrative summary. Only works once the job is `completed` — returns a 400 otherwise.

### List jobs
```bash
curl http://localhost:8000/jobs
curl "http://localhost:8000/jobs?status=completed"
```

## The cleaning pipeline, in order

1. Dedup: exact duplicate rows dropped (the sample CSV has several, e.g. `TXN1009` and `TXN1079` each appear twice verbatim)
2. Dates: `DD-MM-YYYY`, `YYYY/MM/DD`, and `YYYY-MM-DD` all get normalised to ISO `YYYY-MM-DD`
3. Amounts: `$` prefix stripped, parsed as a real number
4. Currency/status: uppercased and trimmed (`inr` → `INR`, `success` → `SUCCESS`)
5. Category: blanks filled with `Uncategorised`, which is what flags a row for LLM classification later
6. txn_id: blank strings converted to actual NULL, not left as empty strings, so the database stores them consistently

## Anomaly rules

- Statistical outlier: amount is more than 3x that account's median transaction. This catches the `TXN2000`–`TXN2004` batch in the sample data, which are an order of magnitude larger than everything else on their accounts.
- **Currency mismatch**: USD currency on a merchant that's domestic-only (Swiggy, Ola, IRCTC, Zomato, Jio Recharge, BookMyShow, HDFC ATM). A USD payment to Zomato doesn't make sense and gets flagged.

A row can be flagged for both reasons at once — the `anomaly_reason` field concatenates them rather than only keeping one.

## LLM usage

Two calls per job, not one per row:

- Classification: every row still marked `Uncategorised` after cleaning gets batched into a single prompt, and Gemini returns a JSON map of `txn_id → category`. Batching matters here both for cost and because the assignment explicitly calls out not to do per-row calls.
- Narrative summary: one more call at the end, fed the aggregated stats (totals by currency, top merchants, anomaly count), asking for a 2–3 sentence summary and a `low`/`medium`/`high` risk rating.

Both calls go through the same retry helper: 3 attempts with exponential backoff (1s, 2s, 4s). If all three fail — bad key, rate limit, network blip the job doesn't fail. The affected rows get `llm_failed: true` and the job still completes with whatever did succeed. This was a deliberate choice: a flaky third-party API shouldn't be able to take down the whole pipeline.

## Known limitations

- `risk_level` and `narrative` come back `null` if the LLM call fails after all retries — by design, not a bug, but worth knowing if you're testing without a valid API key.
- No authentication on any endpoint. Fine for this assignment, not fine for anything real.
- The uploaded CSV is saved to disk inside the container under `/app/uploads/{job_id}.csv` and never deleted. For a 90-row test file that's nothing; at real scale you'd want this in object storage with a retention policy.
- Single worker process with default concurrency. Fine for an assignment-sized file; a production version would need actual load testing to size this properly (more on that in the video walkthrough).

## Database

Schema lives in `migrations/001_initial_schema.sql` and is applied automatically the first time the `db` container starts — Postgres' official image runs anything in `/docker-entrypoint-initdb.d/` on first boot, and `docker-compose.yml` mounts the migrations folder there.

Three tables: `jobs`, `transactions` (FK to `jobs`, cascades on delete), `job_summary` (one-to-one with `jobs`). If you want to inspect data directly:

```bash
docker compose exec db psql -U postgres -d transactions_db -c "SELECT txn_id, date, merchant, amount, currency, category, is_anomaly FROM transactions LIMIT 10;"
```

## Stopping everything

```bash
docker compose down
```

Add `-v` if you also want to wipe the Postgres volume and start from a truly empty database next time:

```bash
docker compose down -v
```
